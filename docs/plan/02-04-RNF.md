# 02.04 — Requisitos Não Funcionais (RNF)

Os Requisitos Não Funcionais definem as características de qualidade e os comportamentos esperados do sistema, independentemente das funcionalidades 
implementadas. Eles descrevem aspectos como desempenho, segurança, disponibilidade, usabilidade, escalabilidade, padrões tecnológicos e 
boas práticas que a aplicação deve seguir. 
Enquanto os RFs definem "o que o sistema faz", os RNFs definem "como o sistema deve se comportar".

## Estrutura dos Requisitos Não Funcionais (RNF)
Cada RNF deve conter:

1. **Descrição** – Define a característica de qualidade exigida do sistema.  
2. **Justificativa** – Explica por que esse requisito é necessário.  
3. **Critérios de Aceite** – Regras objetivas e mensuráveis que permitem validar se o RNF foi atendido.  
4. **Restrições / Considerações Técnicas** – Limitações, padrões obrigatórios ou requisitos tecnológicos.  
5. **Dependências** – Relação com outros requisitos ou componentes que influenciam sua implementação.

Os RNFs descrevem "como o sistema deve se comportar", e não o que ele faz.


## EXEMPLOS
---

## RNF-01 — Performance
- Operações simples devem responder em até **2 segundos**.
- Listagens devem possuir paginação sempre que houver mais de 50 registros.

---

## RNF-02 — Segurança
- Todas as operações devem ser protegidas por HTTPS.
- Senhas devem ser armazenadas utilizando hash seguro (bcrypt ou equivalente).
- Tokens devem expirar.
- Perfis e permissões devem ser aplicados rigorosamente.

---

## RNF-03 — Disponibilidade
- O sistema deve estar disponível 99% no horário comercial.

---

## RNF-04 — Padrões de Código
- Backend deve seguir architecture listada no módulo 07.
- Nenhuma lógica de negócio pode existir em rotas ou controllers.

---

## RNF-05 — Consistência de API
- Rotas devem seguir o padrão REST.
- Respostas devem seguir padrão JSON especificado.

---

## RNF-06 — Persistência
- Banco deve garantir integridade relacional.
- Operações críticas devem utilizar transações quando necessário.

---

## RNF-07 — Escalabilidade futura
- O sistema deve permitir futura evolução para microsserviços.
